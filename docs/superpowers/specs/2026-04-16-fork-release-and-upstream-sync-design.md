# Fork 首发与上游同步设计

日期：2026-04-16

## 概述

本设计用于当前 `sub2api` fork 的首次发布，以及之后与原始项目保持可持续同步。

本次已确认的约束如下：

- 保留项目名 `sub2api`，不做品牌重命名。
- `main` 继续作为 fork 自己的发布主线。
- 首发仅发布到 `GHCR`。
- 与上游保持手工同步，不引入自动同步流程。

目标是先安全发出一版，同时避免为后续合并上游制造不必要的冲突面。

## 当前仓库状态

基于当前仓库观察到的事实：

- 远端目前只有 `origin`，尚未配置 `upstream`。
- 当前工作分支为 `main`，且本地与 `origin/main` 对齐。
- 现有 GitHub 发布工作流位于 `.github/workflows/release.yml`，由 `v*` tag 或手动触发。
- 发布工作流已经支持 `GHCR` 登录，并通过 GoReleaser 构建与发布镜像。
- `SIMPLE_RELEASE` 打开时，工作流会切换到 `.goreleaser.simple.yaml`，仅发布 `GHCR` 的 `linux/amd64` 镜像。
- 当前发布边界仍残留上游身份信息，例如 `Dockerfile.goreleaser` 中的 `org.opencontainers.image.source` 和 `deploy/DOCKER.md` 中的镜像示例。

这些事实说明：当前仓库已经具备首发能力，但在“发布身份”和“用户拉取说明”上仍需做最小修正。

## 目标

- 复用现有发布链路完成 fork 的首次发布。
- 不改项目名，不重构现有 CI/CD 主体逻辑。
- 只修正发布边界上的 fork 身份信息，避免发布产物继续指向上游。
- 为后续手工合并上游建立一个低冲突、可重复执行的流程。

## 非目标

- 不在本次设计中引入自动同步 workflow。
- 不在本次设计中系统性重写 README、安装文档或品牌文案。
- 不修改镜像名 `sub2api`。
- 不要求本次首发同时支持 DockerHub。
- 不将 `main` 回退为“完全贴近上游”的镜像分支。

## 推荐方案

### 方案对比

#### 方案 A：零改动首发

直接沿用现有仓库状态打 tag 发布。

优点：

- 最快。
- 几乎不需要额外修改。

缺点：

- 镜像元数据仍会指向上游仓库。
- Docker 文档示例会继续误导用户拉取上游镜像。
- 首发对外呈现不够干净。

#### 方案 B：最小修补首发（推荐）

保留当前项目结构和发布逻辑，只对发布边界做最小修正后首发。

优点：

- 发版路径短，风险低。
- 与上游的差异面最小，便于后续合并。
- 可以修正镜像归属、拉取示例和仓库来源等关键对外信息。

缺点：

- 部分通用文档和说明仍可能保留上游风格。
- 后续如果希望彻底 fork 化，仍需二次整理。

#### 方案 C：完整 fork 化后再首发

先全面修改文档、品牌、镜像说明和发布描述，再进行首次发布。

优点：

- 对外最完整。

缺点：

- 会显著扩大与上游的差异面。
- 首发前准备周期长，不符合当前“先发一版”的目标。

### 结论

采用方案 B：最小修补首发。

## 设计

### 1. 分支与远端模型

- `main` 作为 fork 的长期产品主线和发布主线。
- `origin` 指向 fork 自己的 GitHub 仓库。
- 新增 `upstream` 指向原始项目仓库，但不改变默认推送远端。
- 后续上游同步直接合并到 `main`，不额外维护一个“纯上游镜像分支”。

这样做的原因是：当前 fork 已经在支付链路上产生持续定制，继续把 `main` 作为自己的主线更符合实际演进路径。

### 2. 首发发布模型

- 首发继续复用 `.github/workflows/release.yml`。
- 首发仅发布到 `GHCR`。
- 首发应走简化发布配置 `.goreleaser.simple.yaml`。
- 通过仓库变量 `SIMPLE_RELEASE=true` 或手动触发时传入 `simple_release=true`，确保工作流只构建 `GHCR` 的 `linux/amd64` 镜像。

这个模型的核心是：不改工作流主体逻辑，只切换到已经存在的轻量发布模式。

### 3. 首发前的最小修改边界

仅允许修改以下两类内容：

- 发布产物中的 fork 身份信息。
- 用户可见的镜像拉取示例。

必须修改：

- `Dockerfile.goreleaser`
  - 将 `org.opencontainers.image.source` 从上游仓库改为当前 fork 仓库。
- `deploy/DOCKER.md`
  - 将镜像拉取与 compose 示例改为当前 fork 的 `GHCR` 地址。

建议一并修改：

- `Dockerfile`
- `deploy/Dockerfile`

这两个文件虽然不直接决定本次 GHCR 自动发布结果，但它们也包含容器源码来源标签。一起修正可以避免本地手工构建继续暴露上游来源。

本次明确不修改：

- README 主体结构
- 项目名
- 业务代码
- 工作流结构
- 发布版本策略

### 4. 首发顺序

首发按以下顺序执行：

1. 在 `main` 上完成最小发布边界修正。
2. 推送到 `origin/main`。
3. 在 GitHub 仓库中确认 Actions 权限和 Packages 权限正常。
4. 确认 `SIMPLE_RELEASE=true` 已生效。
5. 在本地完成针对性的业务验证，重点覆盖支付链路和基础启动链路。
6. 创建发布 tag。
7. 推送 tag，触发 `release.yml`。
8. 等待 `Release` 工作流完成。
9. 在 `GHCR` 中确认镜像包、标签和归属信息正确。
10. 使用实际发布出的 GHCR 镜像进行一次容器拉起验证。

### 5. Tag 与版本策略

为了减少与上游未来版本号的认知冲突，首发 tag 可以采用以下任一方式：

- 继续沿用上游风格的正式语义化版本，例如 `v0.1.114`
- 使用 fork 可识别的后缀版本，例如 `v0.1.114-iuhl.1`

推荐：

- 如果这次发布是对上游某个版本的直接小幅 fork，使用带后缀的 tag，例如 `v0.1.114-iuhl.1`

原因：

- 用户能看出这是 fork 版本。
- 后续合并上游时，不容易和原始项目 release 名称混淆。

### 6. 后续手工同步模型

后续与上游同步时，固定采用以下流程：

1. `git fetch upstream`
2. 切回本地 `main`
3. 执行 `git merge upstream/main`
4. 解决冲突
5. 跑最小回归验证
6. 视情况决定是否发版

约束如下：

- 不使用 `rebase upstream/main` 作为常规同步方式。
- 不把“同步上游”和“立即发版”绑定为同一次操作。
- 不在同步时顺手做无关重构。

这样可以把同步动作限制为低频、显式、可回退的合并操作。

### 7. 冲突处理原则

与上游合并时，优先保护以下内容：

- 支付功能定制。
- fork 的发布边界修正。

其余内容尽量向上游靠拢，尤其是：

- 通用基础设施
- 测试修复
- 非支付业务逻辑
- 通用 CI 修复

这个原则的目的是把长期分叉面压缩到最少，只在确实属于 fork 产品差异的地方维持偏离。

## 错误处理与风险

### 风险：首发 workflow 成功，但镜像归属信息仍指向上游

缓解方式：

- 首发前显式修正 `Dockerfile.goreleaser`。
- 发版后检查 GHCR 包详情中的来源仓库信息。

### 风险：工作流未切到 simple release，发布出不符合预期的产物

缓解方式：

- 发版前确认 `SIMPLE_RELEASE=true`。
- 检查 `release.yml` 的 GoReleaser 参数是否使用 `.goreleaser.simple.yaml`。

### 风险：支付定制在未来与上游支付改动频繁冲突

缓解方式：

- 将 fork 的长期差异尽量限制在支付相关代码和发布边界。
- 每次合并上游后优先检查支付链路。

### 风险：首发过度修改文档，增加未来同步冲突

缓解方式：

- 本次只做最小发布边界修正。
- 其它品牌化工作延后，按需再做。

## 测试与验证

首发前至少验证以下内容：

- 本地业务验证
  - 支付下单主路径
  - 支付结果页
  - 二维码或支付跳转链路
  - 服务基础启动与 `/health` 健康检查

- 发布验证
  - `release.yml` 正常执行
  - `GHCR` 包成功生成
  - 生成的标签符合预期
  - GHCR 包归属当前 fork

- 容器验证
  - 使用发布出的 GHCR 镜像拉起容器
  - 容器启动后健康检查通过

本次不要求为了首发专门新增新的自动化测试体系，但要求至少完成一次真实镜像拉起验证。

## 实施大纲

1. 修正发布边界相关文件中的 fork 身份信息。
2. 设置并确认 `SIMPLE_RELEASE=true`。
3. 校验 GitHub Actions 与 Packages 权限。
4. 完成支付与启动链路的本地验证。
5. 创建并推送首发 tag。
6. 验证发布结果与 GHCR 包。
7. 配置 `upstream` 并记录后续手工同步流程。

## 验收标准

- 首发成功通过现有 GitHub Release 工作流发布到 `GHCR`。
- 发布出的镜像来源和仓库归属指向当前 fork，而不是上游仓库。
- 用户文档中的 Docker 拉取地址不再指向上游镜像。
- `main` 明确作为 fork 的长期发布主线。
- 本地已经配置 `upstream`，具备后续手工同步基础。
- 后续同步流程被明确约束为 `fetch + merge`，而不是临时决定。
